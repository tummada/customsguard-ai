## 🏗️ Phase 1 Summary: Resource Contract & Infrastructure

เป้าหมายของเฟสนี้คือการเปลี่ยน Docker Compose ให้เป็น **"กฎหมายควบคุมทรัพยากร"** เพื่อให้ระบบทำงานได้เสถียรที่สุดบนข้อจำกัด **2 CPU / 8GB RAM** โดยเน้นกลยุทธ์ "High Availability on Low Hardware"

### 📊 ตารางจัดสรรทรัพยากร (Resource Allocation Matrix)

| Service | Role | CPU Limit | RAM Limit | Priority |
| --- | --- | --- | --- | --- |
| **PostgreSQL 16** | Database (RLS Ready) | 0.4 | 1.5 GB | **High (OOM -500)** |
| **Redis 7** | Broker / PubSub / Rate Limit | 0.2 | 512 MB | **High (OOM -500)** |
| **Java Backend** | Core API (GraalVM Optimized) | 0.7 | 1.0 GB | Medium |
| **n8n (Main)** | Workflow UI & Admin | 0.2 | 512 MB | Medium |
| **n8n (Worker)** | AI Logic & JSON Processor | 0.3 | 1.5 GB | Medium |
| **Nginx Proxy** | Reverse Proxy & Static Host | 0.1 | 256 MB | Low |
| **OS Buffer** | System & Background Tasks | **0.1** | **~2.7 GB** | Critical |

---

### 🛡️ กลยุทธ์การบริหารจัดการทรัพยากร (Technical Strategies)

* **Data Sovereignty & Persistence:**
* กำหนด `oom_score_adj: -500` ให้กับ Postgres และ Redis เพื่อป้องกันระบบปฏิบัติการฆ่า Service ข้อมูลทิ้งเมื่อ RAM วิกฤต
* ใช้ **shm_size: 1gb** เพื่อรองรับการทำ Parallel Query ใน PostgreSQL สำหรับงาน Multi-tenant


* **Binary-Free Architecture:**
* ระบบจะไม่เก็บไฟล์ภาพ/วิดีโอในเครื่อง (Disk/RAM) แต่จะใช้ **External S3 (Cloudflare R2/AWS)** ผ่าน Presigned URLs เท่านั้น เพื่อลดภาระ I/O และ RAM


* **n8n Queue Mode:**
* แยกสถาปัตยกรรม n8n เป็น **Main (UI)** และ **Worker (Job)** โดยใช้ Redis เป็นตัวกลาง (Broker) เพื่อป้องกัน UI ค้างขณะประมวลผลงาน AI หนักๆ


* **JVM Native Readiness:**
* บีบ RAM Backend เหลือ 1GB เพื่อบังคับให้ใช้ **GraalVM Native Image** หรือ JRE ที่ปรับจูนมาอย่างดี (MaxRAMPercentage)


* **Disk & Log Governance:**
* บังคับใช้ **Log Rotation (10MB x 3 files)** ผ่าน YAML Anchors เพื่อควบคุมพื้นที่ Disk 100GB ไม่ให้เต็มจาก Log ของ AI



---

### 🏗️ แผนผังการทำงาน (Architecture Overview)

1. **Request Flow:** User -> Nginx -> Backend -> Redis -> n8n Worker.
2. **Data Flow:** Backend (Postgres RLS) -> Object Storage (S3).
3. **Real-time Flow:** Backend -> Redis Pub/Sub -> SSE -> Frontend.

> **Note สำหรับทีม:** "กฎหมาย RAM" นี้ถูกออกแบบมาให้เหลือ CPU Buffer 0.1 เพื่อให้ระบบปฏิบัติการ (Ubuntu) ยังสามารถทำ Memory Swapping และ Maintenance ได้โดยไม่เกิด Kernel Panic
