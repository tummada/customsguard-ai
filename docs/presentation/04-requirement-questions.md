## Requirement Output Checklist

กลับจากประชุมแล้วต้องได้ข้อมูลอย่างน้อย:

- [ ] ขั้นตอนงานจริง (workflow) ตั้งแต่ต้นจนจบ
- [ ] จำนวนใบขน/วัน, รายการ/ใบขน (= ขนาด workload)
- [ ] **โปรแกรมที่ใช้อยู่** (Netbay, TIFFA, ระบบภายใน, Excel)
- [ ] **รูปแบบไฟล์ที่ใช้** (XML schema? EDI format? version?)
- [ ] ประเภทสินค้าที่ทำบ่อย (= HS Code หมวดไหนต้อง priority)
- [ ] Pain points จัดลำดับ (= พัฒนาอะไรก่อน)
- [ ] **ค่า Amendment ต่อเดือน** (= ROI ที่ชัดเจนที่สุด)
- [ ] **ความถี่ใบขนสายแดง** + พิกัดที่มักโดน (= data สอน AI เรื่อง risk)
- [ ] **กระบวนการตรวจงาน** (คีย์ → ตรวจ → ยืนยัน กี่ชั้น?)
- [ ] **ปัญหา iFrame/e-Customs ค้าง** (= ต้องทำ Auto-Save แข็งแรงแค่ไหน)
- [ ] **วิธีดึงเรทแลกเปลี่ยนกรมศุลฯ** (= ต้องทำ API ดึงเรทมั้ย)
- [ ] ความกังวลเรื่อง security / ข้อมูลลูกค้า / ข้อจำกัด IT / PII
- [ ] ราคาที่ยอมรับได้ (pricing signal)
- [ ] จำนวน users ในบริษัท

## หลัก 5 ข้อในการดึง Requirement

1. **ถามเปิด ไม่ถามปิด** — "ทำยังไงบ้าง?" ดีกว่า "ใช้ Excel ใช่มั้ย?"
2. **ขอดูหน้าจอ** — "พี่เปิดให้ดูได้มั้ยว่าตอนนี้ทำงานยังไง?"
3. **Follow the pain** — ถ้าเขาบ่นเรื่องไหน ขุดลึกต่อเรื่องนั้น
4. **จดทุกอย่าง** — แม้จะดูไม่สำคัญ ข้อมูลดิบคือทองคำ
5. **ไม่สัญญา** — "ผมจดไว้ครับ จะกลับไปดูว่าทำได้มั้ย" ดีกว่า "ได้ครับ ทำให้แน่นอน"
